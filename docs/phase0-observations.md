# Phase 0 Observations

## 我們做出了什麼？

- 首頁可以看到 Phase 0 原始資訊列表，並能切到「整理工作台」。
- 工作台會顯示每筆原始資訊的來源、查核狀態、原文與保守安全預設。
- 工作台已有前端記憶體內的整理草稿，可以建立、編輯、刪除與重設；這些草稿不會改寫原始資訊，也不會存成正式整理後資料。

## 哪些資料讓 agent 很容易整理？

- M-010 的原文包含時間、現場更新、物資數量、下一次盤點時間等線索，看起來比較容易整理成候選摘要。
- M-009 說明集合點、限制條件與尚未看到官方公告同步更新，能看出哪些部分仍需要人工確認。
- 這些「容易整理」只代表欄位線索較多，不代表已經是 confirmed / verified 事實。

## 哪些資料讓 agent 亂補或誤判？

- M-001 只有「老雜貨店後面」這種模糊位置，agent 很容易自己補成具體地址或可派工地點。
- M-004 是群組轉述，agent 可能把「還有很多雨鞋」當成現場庫存事實，但原文沒有查核。
- M-005 的來源型態像公告，但原文說不知道是哪一天、也不知道是不是官方公告；不能只看 `sourceType` 就當成官方已確認。
- M-012 來電者不在現場，agent 可能把「疑似需要藥品協助」轉成任務，但原文連親友目前位置都無法確認。

## 哪些資訊不能直接變成任務？

- M-001 缺少可確認地點、需求數量細節與現場窗口，不能直接派十幾個人。
- M-006 對集合點是否適合停留有互相衝突的說法，不能直接建立集合任務。
- M-008 只說 A 區先不要再派人，但沒有說原因；若原因是道路危險或任務完成，後續處理會完全不同。
- M-011 涉及長者住家與同意公開完整地址的問題，不能直接公開或派工。
- M-012 涉及疑似藥品需求與不在現場的家屬轉述，必須先確認當事人狀況。

## 哪些資訊的操作者不是當事人？

- M-011 是現場志工代長者轉述，志工不是需求本人，還有公開完整地址同意問題。
- M-012 是外地家屬來電，來電者不在現場，也不能確認親友目前位置。
- M-004 是群組中「有人說」的轉述，不是庫存管理者或現場確認者。

## 我們剛剛偷偷假設了什麼？

- 假設「來源型態」可以代表可信度；但 `sourceType` 只代表資料從哪裡來，不代表已查核。
- 假設原文中的地名描述足以讓志工找到位置；實際上 Phase 0 不應補真實地址或地圖。
- 假設需求描述等於可派任務；但任務還需要當事人同意、時間、地點、安全與窗口等資訊。
- 假設 agent 產出的候選分類可以直接採用；實際上它只是協助思考的草稿。

## 如果要讓這個工具變可靠，下一步需要什麼？

- 需要更明確的資料格式，把原文、來源、查核狀態、候選判斷、人工確認與不能行動原因分開。
- 需要人工確認角色，例如現場窗口、資料整理者、當事人同意確認者。
- 需要狀態規則，避免 `needs_review` 或 `unverified` 被畫面或 agent 說成已確認。
- 需要保存人類修正紀錄，讓下一位協作者知道哪些 agent 判斷已被質疑。

## 我們需要更清楚資料格式的地方

- 來源型態與查核狀態要分開，不能讓 `official_notice` 直接等於已確認。
- 需要欄位記錄「資訊操作者是否為當事人」與「是否有公開個資或地址的同意」。
- 需要欄位記錄「不能直接變成任務的原因」，例如地點不明、時間過期、說法衝突、涉及同意或安全。
- 需要欄位記錄「agent 推測」和「原文可直接看出的資訊」的差異。

## 我們需要人工確認的地方

- M-005 需要確認截圖日期、公告來源與目前是否仍有效。
- M-006 需要確認集合點是否安全，以及哪個回報比較新。
- M-009 需要確認官方公告是否更新，以及集合點限制是否仍有效。
- M-010 需要確認盤點時間後庫存是否改變。
- M-011 需要確認長者需求、地址公開同意與可聯絡窗口。
- M-012 需要確認當事人位置、藥品需求是否存在，以及是否適合建立任務。
