# 需求取捨決策

> 這份文件由 Codex 先依 Release 01 訪談結果產生草稿。最後必須由學員刪除、修改或補充；不要讓 AI 替你做最後決策。

## 我決定 v1 優先服務的使用者

- [ ] 回報者
- [x] 資訊整理者
- [ ] 行動者

原因：

- 三個 persona 的回饋都指出同一個風險：使用者容易把「看起來整理過」誤認為「已確認」或「可以行動」。因此 v1 應先幫資訊整理者把原文、整理草稿、AI 推測、人工判斷與查核狀態分開，而不是先追求快速派工。
- Phase 0 原型目前最主要的價值，是讓下一位協作者看懂混亂原始資訊中的缺漏、疑點與不能直接變成任務的原因。這個價值最接近資訊整理者的工作，而不是回報者的輸入流程或行動者的現場任務流程。
- 若直接優先服務行動者，畫面很容易被誤解成「現在可以照做」的派工工具；但目前資料仍有轉述、來源不明、時間不明、地點不清、當事人未確認等問題，不適合被包裝成行動指令。
- 若直接優先服務回報者，v1 會需要設計輸入表單、隱私提醒與回報流程，容易偏離這一階段「整理既有 Phase 0 原始資訊」的任務範圍。
- Release 01 是需求分析階段，不是實作階段。這個選擇先作為 v1 方向草稿，仍需要學員在小隊討論後確認或修改。

## 這個使用者最想完成什麼

- 快速看出每筆原始資訊目前能不能初步整理，或只能先標記疑點。
- 分清楚原文、AI 推測、人工判斷與整理草稿。
- 留下為什麼不能直接變成任務的理由，讓下一位協作者不要誤判。

## 這個使用者最怕什麼錯誤

- 模糊地點、轉述或過期資訊被整理成可派工任務。
- AI 補出原文沒有的地址、數量、角色或確認狀態。
- 下一位協作者只看到摘要，沒有看到原文與疑點，就以為資料已經確認。

## 我今天暫時不處理的需求

- 暫時不把工作台改成行動者派工畫面。
- 暫時不設計真實回報表單或真實救災流程。
- 暫時不處理外部地圖、外部查核、後端、資料庫或 runtime LLM API。

## 我不採用的 AI 建議

- AI 建議：把三個 persona 的回饋直接轉成 v1 功能清單並開始改前端。
  - 不採用原因：Release 01 是需求分析階段，不應修改 `src/`；需求取捨需要人類確認。
- AI 建議：用 AI 自動判斷哪些資訊可以行動。
  - 不採用原因：這會把未確認資訊包裝成決策，違反不做真實救災判斷的邊界。

## 必須保留人工確認的判斷

- 資訊取得方式不等於可信度，也不等於官方確認。
- `needs_review` / `unverified` 不能顯示成 `verified` / `confirmed`。
- 涉及個資、地址、藥品、長者需求、當事人同意或現場安全的內容，必須保留人工確認。
- AI 產生的摘要、候選分類與下一步建議，都必須能被人類質疑、修正或拒絕。

## 今天明確不做什麼

- 不修改 `src/` 或把 Release 01 訪談結果直接做成新功能。
- 不新增後端、資料庫、外部 API、地圖或 runtime LLM API。
- 不補真實地址、電話、人物、社群資訊、政府資料或新聞。
- 不把訪談草稿當成真實使用者結論。

## 我仍不確定的地方

- 待學員確認：v1 是否只服務資訊整理者，或要同時保留行動者能理解的安全提示。
- 待學員確認：「不可行動」要做成單一醒目標籤，還是拆成多個原因。
- 待學員確認：小隊討論後是否要調整優先服務對象。

## Release 02 流程設計後的確認

- 暫時維持 v1 優先服務「資訊整理者」的決策。
- 流程從 Phase 0 原始資訊開始，不從整理後資料或派工任務開始。
- 候選整理結果只能是待確認輸出，不代表已確認，也不代表可以行動。
- 新增一個明確分支：即使資訊看起來足夠，只要含有轉述、來源不明、地點不清、時間不明、涉及個資或可能影響現場行動，就不得自動轉成可派工任務。
- 待學員人工確認：`docs/flow.md` 的 Mermaid 是否能正確預覽，以及流程是否符合小隊共識。
